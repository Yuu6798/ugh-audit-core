2. P2 selective cache (proposition のみ、segments は bypass) + hard cap

7. P2 startup invalidation (lazy load 前の呼び出しが no-op にならないように)

9. P2 per-process tmp file (fixed `.tmp` 名での atomic rename 衝突)
10. P2 共有モデル bounded retry (一過性失敗での permanent disable 防止)
11. P2 flush startup preservation (lazy load 前 flush で disk wipe 防止)

## 成功パターン

### Caching layer の段階的 hardening
Codex レビューが PR #60 で 11 件連続して状態管理の穴を指摘 → それぞれ 1 commit で修正 → 修正が fallout を生むと次の commit で対応、というパターンで非常に clean に進んだ。caching layer のような state machine は single review では不十分、複数ラウンドが必要。

### Before/after test pattern for regression
Bug 修正時に、ユーザー指摘内容をそのままテスト名にする (例: `test_invalidate_at_startup_purges_stale_disk_file`, `test_sbert_rerank_recovers_from_stale_cache_dim`) ことで、将来の同種 regression を検出できる。

### Task-stratified metric comparison
単純な before/after は task distribution の変化に confound されるため、task classifier で層別化してから比較するのが必須。Phase 1 でこれを実証。

### 外部レビューを活用した proxy metric の品質向上
自作 metric は自分では bug を検出できない (self-referential loop) → Codex のような external actor を介在させることで Phase 1 実行前に 7 件の bug が修正された。

## 設計判断

### File lock を入れない (永続 cache の concurrent write)
研究段階の HA48 scale + 少数プロセス運用に対して portalocker / fcntl は過剰と判断。**reload-then-merge + invalidation_pending フラグ** で 90% の lost-update を緩和し、残る race window は許容。production scale 移行時は portalocker 追加を検討する旨を docs/cascade_design.md に明記。

### Fingerprint-based cache key を入れない
Codex P1 r3067145596 の shape guard 指摘に対して、「SBert weights hash を key に含める」方式は起動時の全 tensor ハッシュが過剰と判断。**layered shape guard** (cache 層の `_cache_embed_dim` + caller 側の proactive shape check + graceful invalidation) の方が軽量かつ十分。

### Phase 1 は凍結、Phase 2 以降は次回以降
Phase 1 は single session での proof of concept。正式な research claim には Phase 2 ablation / Phase 3 task diversity / Phase 4 cross-model が必要。次セッションは意味監査ツール本体 (detector / semantic_loss / audit.py) のタスクに戻るため、Phase 1 は凍結してドキュメントに再開マーカーを残した。
