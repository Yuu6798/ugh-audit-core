## 成功パターン

### 1. git blame による独立性の確定情報化
同時並行 PR の scope 判断で「物理的に離れている」という主観的表現を排除し、
`git blame -L <range>` で「最近 N commits で同一 PR に触られていない」
という確定情報で記述する。#2 critique で user に指摘されたが、実際に
運用すると G-3 / G-4 / G-5 / rename PR 4 本の並列走行判断が即決できた。

### 2. 実装確認の反証時エスカレーション
プロンプトに「前提 X が実装と一致しない場合は PR を draft に戻し、
user にエスカレーションして作業停止」を明記する運用。docs を silent
adjust で実装に辻褄合わせすると査読時に破綻するため。G-3 追記 1 で
初導入し、G-2 / G-5 / rename プロンプトにも展開。

### 3. 環境分離前提での数値相互検証
Claude が動く Linux サンドボックスと user の Windows ローカル (`...lazyload/`)
は別環境。Codex の PR 出力を Claude 側では実行 verify できない。代わりに
事前に手動実行した数値 (q001 良質 ΔE=0.0651) と Codex 報告値の一致を
もって信頼する運用。

### 4. Sprint spec の fact-check を前提化
spec は old snapshot 時点の記述が多く、現実装と乖離していることが多い。
プロンプト化前に `grep -rn`, `git log --oneline`, `git blame` で
**現 repo 状態を確認してから** scope を決める。この手順を踏まなかった
場合、Codex が「直すべきものがない」で停止する or 誤認 spec に従って
壊れた修正を入れるリスクがある。

### 5. Verdict 多様性 ≥ 2 アサートによる回帰テスト
tests/test_examples.py で具体的な verdict 値 (accept / regenerate) を
個別に assert するだけでなく「3 ケースで verdict が 2 種類以上出ている
こと」を assert する。校正変更で ΔE 絶対値が動いても「パイプラインが
定値を返していないこと」は保証される。

## 設計判断

### AuditCollector の broken behavior は sprint 外
`ugh_audit/collector/audit_collector.py` の `AuditCollector.collect()`
が `Evidence(question_id="unknown")` を直接 calculate() に渡し常に
degraded を返す問題を発見。しかし `tests/test_collector.py` の 4 件
(L32-86) が **この degraded 動作を仕様として lock** している:
- `test_collect_degraded_not_saved_to_db`
- `test_collect_batch_degraded`
- `test_collect_degraded_returns_none_values`
- `test_session_context_manager_degraded`

「バグ」ではなく「意図的仕様」扱いなので、修正には test 4 件書き換え +
公開 API 動作変更が必要。B スプリント送り確定。

### PR #89 の demo fix は audit.py 経由、AuditCollector 経由ではない
G-1 の修正ルートとして、AuditCollector ではなく `audit.py` の `audit()`
関数を直接呼ぶ構成を採用。理由: AuditCollector 自体が上記バグを抱えて
おり、使うと demo が正しく動かない。`_load_question_meta` も `audit.py`
からインポート。

### rename PR を G-4 の前に merge
G-4 に L185 典型文修正を fold せず、rename PR (O-3 解決) を独立 micro-PR
として分離。理由: scope 観点 (docs vs data layer rename) が違い review
が散漫になる、rename merge 後に G-4 は新 filename 前提で書ける。

### `tests/test_examples.py` は `working behavior を lock` する構造
`test_collector.py` が broken behavior を lock しているのと対称的に、
example の「動作する状態」を CI で恒常的に保証する構造を導入。これで
将来の audit.py リファクタで demo が再 break しても検知できる。

### 子孫 JSONL の source_file フィールドは新名上書き
rename PR で `q_metadata_structural_*.jsonl` 3 ファイルの `source_file`
値を旧名から新名に上書き更新。歴史保持より現 canonical path を指す方が
