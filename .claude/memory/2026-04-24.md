## 成功パターン

- **`detect_fn` を callable param として渡す設計**: pipeline.py を pure
  computation に保ちつつ、server.py / mcp_server.py の module-level
  `_HAS_DETECTOR` / `_detect` を保持できる。test の
  `monkeypatch.setattr(server_module, "_HAS_DETECTOR", False)` や
  `monkeypatch.setattr(server_module, "_detect", raising_fn)` が
  そのまま動く。分離と backward compat の両立
- **parity test in parametrize**: 同一入力に対する REST/MCP の
  field-by-field 一致を parametrize で複数ケース回すことで、将来 片方に
  patch が入った時に即座に drift を検知できる。今回の共通化の意義を
  test で固定できた
- **`git rebase --onto <upstream> <skip-until> HEAD` パターン**: squash
  merge 後のブランチ整理の標準手。`reset --hard + cherry-pick +
  force-push` より非破壊的で、destructive flag を避けられる。
  「`<skip-until>` 以降の commit のみを `<upstream>` の上に再生する」
  と覚えればよい
- **Codex P2 指摘への surgical 対応**: `detector_failed` フラグ導入 +
  `/audit → /api/audit` 5 箇所修正 はいずれも小規模な surgical fix で
  closure。test の契約を強化する方向 (`assert
  "core_propositions_missing" not in data["degraded_reason"]`) で指摘の
  意図を明文化できた

## 設計判断

- **Option B (段階的非推奨) を採用**: AuditCollector は public API として
  空手形 (verdict="degraded" 固定) だったが、外部 consumer の存在可能性を
  考慮し、即削除ではなく 1 リリース buffer を挟む。v0.5 で物理削除。
  test_collector.py は「現状追認 snapshot」ではなく「DeprecationWarning
  契約 test」として再定義
- **`auto_generate_meta` 導入禁止原則**: 将来の AuditCollector 再建時も
  含めて、question_meta = 監査基準であり自動生成は「計測器層」と
  「判定層」の境界を曖昧にする。決定的監査の設計思想と衝突するため導入
  しない。例外として決定的生成 (rule-based) のみ許容、LLM 生成は禁止
  (既存 server.py の `auto_generate_meta` は LLM 生成だが、これは合意
  形成後の既存 feature なので grand-father。新規 programmatic API には
  導入しない)
- **`degraded_reason` は silent にしない**: fail-safe と silent failure
  は別物。API 契約として理由開示を必須とする。`detector_unavailable` /
  `detector_error:<type>` を明示的に enum に追加
- **schema_version 2.0.0 → 2.1.0 (additive)**: 新しい enum 値は unknown
  string として既存 consumer で無視可能なため breaking change ではない。
  minor bump で新 code path の存在を signal
- **detector_failed vs detection_skipped の区別**: detector fault と
  metadata 問題は retry / alerting の routing が違う故障モード。呼び手が
  切り分けられるよう、両 reason の append を `detector_failed` で guard
- **共通化の timing**: 論文提出前なので risk 最小の surgical 運用を予定
  していたが、user 判断で長期 PR を今日着手 → parity test 5 件で drift
  防止を仕込めたため結果的に低 risk で統合できた
