## 成功パターン

### 1. Codex review ループからの scope pivot
PR #99 で round 1-2 時点で「全ての数値閾値」という promise は運用不能と
判断し、「tunable な主要閾値」に scope narrow + 対象外 (severity state
transition, 内部合成式の定数, 制御定数) を explicit 列挙する構造に pivot。
以降 round 3-8 は scope 内の個別抜けを埋める作業になり、promise が持続
可能な範囲に収まった。「全を謳う索引は必然的に stale 化する、narrow +
判断基準の明文化が最適」は汎用化可能な learning。

### 2. runtime を single source of truth 原則
PR #99/100 で確立。設計 doc (markdown) の数値は校正イベント後に stale
化しやすく、runtime (`.py`) が真値を保持する。閾値索引 `docs/thresholds.md`
の出典列を runtime コード直参照 (`semantic_loss.py:DEFAULT_WEIGHTS`
`ugh_calculator.py:C_BIN_THRESHOLDS` 等) に統一することで、他 doc の
stale が索引経由で検出可能になる構造が生まれた。PR #100 はこの構造を
逆利用して 3 doc を一気に同期。

### 3. PR per concern の厳格分離
本日は 7 PR を並列 / 連続運用。chore/repo-hygiene, refactor/golden-seed-external,
docs/grv-multilingual, docs/thresholds-index, docs/lsem-weight-sync,
ci/weekly-sbert-regression 等、concern 別に branch を切ることで Codex
レビューの混線を回避し、review 負荷を最小化できた。

### 4. E2E install verify (wheel 同梱保証)
PR #97 round 3 の Codex P1「`examples/seed_references.json` は非 editable
install で欠損」を、`python -m build --wheel` → クリーン venv 作成 →
`pip install <wheel>` → 空 cwd で import の手順で実測 verify。指摘の
「will always miss」を裏付け、seed を package 内に移動 + package-data
追加で修正したことを E2E で confirm。

### 5. scope narrow + 具体追記のハイブリッド
promise を narrow するだけでは指摘された legitimate な項目が scope 外と
なるリスクがある。narrow + 指摘項目の追記を同時に行うことで、両方の
要件 (持続可能な promise / 個別抜けの補修) を満たせる。PR #99 round 2
で初採用、以降の round でも同方針を維持。

### 6. LOO-CV 補正の文書化で誤読予防
`semantic_loss.py:35` のコメント行には full-sample 最適 `L_P=0.425` が
書かれており、同 39-47 の DEFAULT_WEIGHTS は LOO-CV 補正後の `L_P=0.27`。
両者を混同した round 4 の自滅を受けて、導出根拠パラグラフで「full-sample
最適 vs LOO-CV 補正後 runtime」の区別を explicit にし、`L_P:L_F:L_G ≈
0.41:0.30:0.91 を正規化」の履歴も添えた。次の編集者が同じ誤読をしない

## 設計判断

### seed JSON は package-internal に配置
当初 `examples/seed_references.json` (reviewer 当初指定の場所) に置いたが、
Codex P1 指摘で非 editable install 時に wheel に同梱されないことが判明
→ `ugh_audit/reference/seed_references.json` に移動 + `[tool.setuptools.package-data]`
に追加。`DEFAULT_SEED_PATH = Path(__file__).parent / "seed_references.json"`
で runtime 解決。reviewer 指示の文字通りの場所より**動作する状態**を優先。

### fugashi/ipadic extras 削除は事前に docs で可視化 → cleanup
PR #98 で grv 多言語節に「extras 宣言されているが repo 内 Python で
import 0 件」と明記、PR #100 で実際に `[ja]` 削除 + `[full]`/`[server]`
から該当 2 行削除。2 段階で扱うことで reviewer の「未使用依存の残置」
指摘を事前 address してから cleanup を実行し、scope の正当性を確立。

### 閾値索引の scope narrow は運用契約としての promise 簡素化
PR #99 round 2 で「全閾値」→「tunable な主要閾値」に pivot したのは
単なる逃げではなく、**索引が実務で同期され続けるための条件**。severity
state transition や内部合成式の定数を全て載せると、変更のたびに索引
更新が義務化され、運用が停止する。scope narrow + 対象外の判断基準
明記 (設計 doc での命名 / 校正スクリプトでの探索対象) で index が
持続可能な promise になる。

### weekly CI は fast path と別 workflow
`ci.yml` は push/PR trigger で Python 3.10/3.11/3.12 マトリクス + dev extras
(SBert 非同梱、7 件 skip)。`ci-weekly.yml` は schedule/dispatch + Python
3.11 単独 + `.[dev,server]` extras (SBert 同梱) で skip 7 件も実行。
目的分離により CI 全体の可読性と維持可能性を確保。Claude サンドボックス
では workflow 実行 verify 不可のため、merge 後ユーザーが手動 dispatch で
初回 verify する必要あり。
