# Session Summary — 2026-04-14-2

## 修正・訂正

- **`.dockerignore` が真の根本原因だった**: `experiments/` が `.dockerignore` に含まれていたため、Docker イメージに `experiments/meta_generator.py` が存在しなかった。`PYTHONPATH=/app` を追加しても、`pip install` で wheel に含めても、そもそもファイルがコピーされていなかった。3つの PR (#72, #73, #74) を経て最終的に `.dockerignore` の 1 行削除で解決
- **`except Exception: pass` が原因特定を妨げていた**: `logger.exception()` に変えた瞬間に `ModuleNotFoundError: No module named 'experiments'` が Railway ログに出現。これがなければ `.dockerignore` の問題は発見できなかった
- **`actually_filled` の判定バグ**: `[] is not None` → `True` で空リストを "filled" とみなしていた。`_is_field_filled()` ヘルパーを導入（リスト型は空で unfilled、文字列型は None のみ unfilled）
- **`trap_type=""` の扱い**: truthiness 判定に変更した際、`trap_type=""` が unfilled 扱いになる回帰。レビューで指摘され修正

## 成功パターン

- **logging 修正を最優先**: silent `except` をまず可視化することで、後続の調査が劇的に効率化された
- **本番エンドポイントを直接叩いて検証**: curl で MCP エンドポイントに JSON-RPC を送り、ローカルとの挙動差を即座に確認
- **レビュー指摘の段階的対応**: chatgpt-codex-connector のレビューが `metadata_source` の来歴問題を段階的に深掘りし、最終的に `META_SOURCE_FALLBACK` + degraded 強制 + `is_reliable=false` + null 化 + `core_propositions` 由来判定まで整備された

## 設計判断

- **`META_SOURCE_FALLBACK = "fallback"` を新設**: `inline` / `llm_generated` / `none` に加え、LLM 未使用のヒューリスティック結果を区別する専用ソースを追加
- **フォールバック時の degraded 強制**: `metadata_source=fallback` のとき `mode=degraded`, `verdict=degraded`, `is_reliable=false`, `C/delta_e/quality_score=null` に強制。analytics 汚染を防止
- **`core_propositions` 由来でゲート**: inline meta に `core_propositions` がある場合、フォールバックは `trap_type` 等の optional フィールドのみ補完 → degraded にしない。`core_propositions` 自体がフォールバック由来のときだけ degraded
- **`_fallback_meta` で質問文を命題として使用**: `core_propositions: [question]` + `_is_fallback: True` マーカー。LLM 不可時でもパイプラインは完走するが、`metadata_source=fallback` → degraded で信頼性は明示的に低い
